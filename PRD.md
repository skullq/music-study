# 🎵 Music Visualization Web Service - PRD (제품 요구사항 정의서)

## 1. 프로젝트 개요 (Overview)
본 프로젝트는 사용자가 선택한 음악적 요소(스케일, 3화음, 텐션 코드, 분수코드 등)를 오선지, 피아노 건반, 기타 지판에 실시간으로 시각화하고 사운드로 재생하는 **인터랙티브 음악 이론 학습 웹 애플리케이션**입니다. 시각적 정보와 청각적 피드백을 동기화하여 직관적인 학습 경험을 제공하는 것을 목표로 합니다.

## 2. 주요 기능 요구사항 (Key Features)
* **통합 시각화 (Multi-instrument Visualization):**
  * **오선지 (Staff):** VexFlow를 활용하여 스케일 간격(온음/반음), 로마숫자 표기(Diatonic), 정확한 임시표 및 코드 네임 렌더링.
  * **피아노 (Piano):** 동적 DOM 생성을 통한 3옥타브 건반, 활성화된 음표 하이라이팅.
  * **기타 지판 (Guitar Fretboard):** 외부 라이브러리(ChordJS 등) 의존 없이 자체 구현한 순수 SVG 그래픽을 활용하여 타브 악보 형태의 운지법 및 스케일 지판 매핑.
* **인터랙티브 오디오 (Web Audio API):**
  * 오선지, 피아노, 기타 지판의 음표 클릭 시 실제 주파수 기반의 합성음(Oscillator) 재생.
  * 단일음 재생(Arpeggio) 및 다중음 동시 재생(Chord) 분기 처리.
* **동적 음악 데이터 연산:**
  * 기초 스케일 및 다이아토닉 화음 쌓기(3화음, 7화음).
  * 9, 11, 13화음 등의 고급 텐션 코드 및 서스펜디드(sus), 분수코드(Slash) 연산 및 매핑.

## 3. 아키텍처 및 기술 리뷰 (Architecture & Tech Review)

### 3.1 기술 스택
* **Frontend:** HTML5, CSS3, Vanilla JavaScript, VexFlow (악보 렌더링)
* **Backend:** Python 3.x, FastAPI, music21 (초기 데이터 제공)

### 3.2 프론트엔드 프레임워크 vs Vanilla JS 리뷰
현재 코드는 React, Vue, Svelte 등의 모던 프레임워크를 사용하지 않고 **Vanilla JavaScript**로 구축되어 있습니다.
* **장점:** 
  1. **가벼움과 직관성:** Webpack, Vite 등의 복잡한 빌드 파이프라인 없이 즉시 실행 및 수정이 가능합니다.
  2. **DOM/Canvas 직접 제어:** VexFlow(Canvas/SVG)와 같이 DOM 객체를 직접 다루고 그리는 라이브러리들은 Virtual DOM을 거치는 프레임워크보다 Vanilla JS에서 제어하는 것이 성능 오버헤드가 적고 통합이 훨씬 수월합니다.
* **단점 및 개선점:** 
  현재 `window.activeStaffMidis`, `window.currentStaffNotesData` 등 전역 객체를 통한 상태 관리(State Management)가 이루어지고 있습니다. 향후 앱의 규모가 더 커져 악기별 렌더링 상태가 복잡해진다면, 상태 추적이 어려워질 수 있으므로 이때는 상태 관리 패턴이나 경량 프레임워크(예: Alpine.js) 도입을 고려할 수 있습니다. 하지만 현재 요구사항 범위 내에서는 최적의 성능과 유연성을 보여줍니다.

## 4. 핵심 엔지니어링 결정: 프론트엔드 화음 연산의 이유

본 프로젝트의 가장 중요한 구조적 전환은 **"화음(Chord) 구성음 계산 로직을 Backend(Python/music21)에서 Frontend(JavaScript)로 이관한 것"**입니다. (`api.js`의 `generateChordData` 함수)

### 이유 1: `music21` 라이브러리의 한계 및 복잡성 우회
백엔드에서 사용하는 `music21`은 클래식 음악 분석에 매우 강력한 도구이지만, 현대 실용음악이나 재즈에서 주로 사용하는 **텐션 코드(9th, 11th, 13th), Add 코드, 분수코드(Slash Chords)**를 일관성 있게 파싱하고 생성하는 데 구조적인 한계가 있습니다. 백엔드에서 이를 억지로 처리하려면 예외 처리 코드가 기하급수적으로 늘어나며 정확도가 떨어집니다.

### 이유 2: 데이터의 100% 통제 및 시각화 정확성 보장
악보, 피아노, 기타는 수학적인 Interval(음정 간격)에 의해 결정됩니다. 프론트엔드에서 음정 배열(예: `major_13: [0, 4, 7, 11, 14, 17, 21]`)을 직접 하드코딩하여 기준 미디(MIDI) 노트에 더하는 방식을 채택함으로써, 화면에 렌더링되는 시각적 데이터의 **정확도를 100% 보장**할 수 있게 되었습니다. 또한, 분수코드의 베이스 음 계산도 JS 함수로 동적 처리하여 뛰어난 확장성을 확보했습니다.

### 이유 3: UI 반응성(Latency) 극대화 및 서버 부하 감소
코드를 선택할 때마다 서버로 HTTP 요청을 보내 응답을 기다리는 대신, 프론트엔드 메모리 상에서 즉시 연산하여 렌더링하므로 **Zero-Latency(지연 시간 없음)**의 쾌적한 UX를 제공합니다. 백엔드는 복잡한 12음계 스케일 연산에만 집중하게 하여 시스템 전체의 부하를 크게 줄였습니다.

## 5. 백엔드 음악 엔진(music21)의 역할과 유지 이유 (중요 가이드라인)

프론트엔드에서 화음 연산을 처리하게 되었음에도 불구하고 백엔드(`music_engine.py`)의 **`music21` 라이브러리를 유지해야 하는 핵심 이유와 역할**은 다음과 같습니다. 향후 리팩토링 시 아래 사항을 반드시 숙지해야 합니다.

### 5.1 `music21`의 현재 역할: 스케일(Scale) 연산의 심장
* **복잡한 음계 및 조표 규칙 계산:** 사용자가 `F# Minor`나 `Eb Major`를 선택했을 때, 조표(Key Signature) 규칙에 맞는 정확한 임시표(♯, ♭)와 구성음을 계산하는 학술적 역할을 전담합니다.
* **다이아토닉 화음(Diatonic Chord) 쌓기 기반 제공:** 스케일의 각 음을 기준으로 3도 간격으로 쌓아 올리는(Triad, 7th) 다이아토닉 화음의 기본 피치(Pitch) 배열을 계산하여 프론트엔드에 공급합니다.

### 5.2 한계점 (화음 연산에서 배제된 이유)
* `music21`은 전통적인 클래식 음악 분석에 특화되어 있습니다. 때문에 현대 대중음악의 **확장 텐션(9th, 11th, 13th), Add 코드, 분수코드(Slash Chord)**를 파싱하고 동적으로 생성하는 데에는 구조적 한계와 일관성 부족 문제가 존재합니다. 이 부분이 화음 생성 로직을 프론트엔드로 이관한 결정적 계기입니다.

### 5.3 라이브러리 제거 시의 치명적 리스크 (Risk)
* `music21`을 백엔드에서 완전히 제거할 경우, 조표에 따른 ♯/♭ 표기 규칙, 이명동음(Enharmonic) 처리, 스케일별 음정 간격 등 **방대하고 예외가 많은 음악 이론 로직을 프론트엔드(JS)나 백엔드(Python)에서 처음부터 다시 하드코딩해야 합니다.**
* 이는 막대한 개발 비용을 초래하며 버그 발생 확률을 기하급수적으로 높이므로, **스케일 연산 영역에서 `music21`은 절대적으로 유지되어야 합니다.**

## 6. 기타 운지법 데이터베이스 활용 리뷰 (`chord-collections.js`)

기타 지판 렌더링을 위한 방대한 코드(Chord) 운지법 데이터는 프론트엔드에서 연산하기에 무리가 있으므로, 외부 오픈소스인 T-vK/chord-collection의 데이터를 채택하여 로컬 정적 파일로 구축했습니다.

### 6.1 데이터 구조 및 역할
* **위치:** `static/js/data/chord-collections.js`
* **역할:** 수천 개의 기타 코드 폼(포지션, 손가락 번호)을 JSON 형태의 전역 객체(`CHORD_COLLECTION`)로 제공하는 **인메모리(In-memory) 데이터베이스** 역할을 수행합니다.
* **특징:** `{ "C": [ { "positions": ["x","3","2","0","1","0"], "fingerings": [...] }, ... ] }` 형태로, 한 코드당 여러 개의 운지법(Variation)을 배열로 가집니다. 배열의 첫 번째 인덱스(`[0]`)는 가장 대중적인 기본(Standard) 폼을 나타냅니다.

### 6.2 프로젝트 내 데이터 처리 파이프라인 (Code Review)
1. **초기 적재 (`index.html`):** 로딩 속도 최적화 및 의존성 주입을 위해 메인 DOM 렌더링 전에 `<script>` 태그를 통해 전역 변수로 가장 먼저 로드됩니다.
2. **데이터 매핑 (`api.js`):** 
   * 사용자가 선택한 조합으로 완성된 코드 네임(예: `Cmaj7`, `C/E`)을 Key 값으로 사용하여 `CHORD_COLLECTION[chordName]`을 조회(Lookup)합니다.
   * O(1)의 시간 복잡도로 즉시 데이터를 가져오며, 사용자 UI의 '검색 옵션(단일/전체 뷰)'에 따라 배열의 첫 번째 요소(`[0]`)만 넘길지, 전체 배열을 넘길지 필터링합니다.
3. **SVG 동적 렌더링 (`guitarRenderer.js`):** 
   * 전달받은 객체의 `positions` 배열을 파싱하여, 숫자인 경우 해당 프렛 위치에 마커를 찍고, `x`인 경우 개방현 뮤트 표시를 하는 등 자체 구현한 순수 SVG 타브 악보 그래픽으로 동적 변환합니다.
